## Context

The Atomicly repo currently has exactly one deployment pipeline: `.github/workflows/ci-cd.yml`, triggered on push to `master`. This means:

- Every infra change has to be merged to master before it can be tested end-to-end on Azure.
- Every Playwright failure that depends on real Azure infrastructure (Front Door routing, Postgres connectivity, OIDC, Key Vault references) is discovered post-merge.
- A broken Bicep template, a typo'd App Service setting, or a missing Key Vault role assignment takes down the only running environment.
- There is no safe way to validate a deploy-pipeline change itself without merging it to master.

The recently-shipped `cost-optimize-azure-infra` change makes per-PR previews newly affordable: with `containerApp.bicep` already authored, scale-to-zero Container Apps + B1ms Postgres makes an idle preview cost roughly $0.55/day instead of the ~$2.50/day an App-Service-backed preview would cost.

The constraint shaping every decision below is that **the agent owning this change has read-only Azure credentials**. Every Azure mutation has to ride through GitHub Actions; nothing can be clicked in the portal. That naturally pushes the design toward declarative IaC + tag-based reaping, which is what we wanted anyway.

## Goals / Non-Goals

**Goals:**

1. Every PR that opens against master gets its own fully isolated Azure deployment, automatically.
2. The PR check is red unless Playwright is green against the **live preview**, not localhost.
3. Every preview is torn down when the PR closes (merged, abandoned, or force-closed). A daily safety-net reaper deletes anything older than 24 h regardless.
4. Cost of a forgotten preview, with the reaper working, is bounded at < $1/day.
5. Master CI/CD is not edited as part of this change.
6. No third-party preview-env service. Same Azure stack as production, smaller SKUs.
7. No Front Door / WAF / Cloudflare on previews. Those are public-surface controls; previews are private testbeds.

**Non-Goals:**

- Custom DNS, vanity URLs, or shareable preview links to non-collaborators. The default `*.azurecontainerapps.io` hostname is fine.
- Multi-region previews. Australia East only, same as dev.
- Cross-PR isolation strong enough for adversarial multi-tenancy. Previews share a subscription; we trust the GitHub-actions identity boundary.
- Persistent preview data. Each preview gets a fresh Postgres with migrations applied; no seed data, no shared state.
- Editing `app/` or `lib/` to support previews. The application is preview-environment-agnostic.

## Decisions

### Decision 1: Naming scheme — derived deterministically from `<pr-number>` and `<short-sha>`

| Resource | Name template | Rationale |
| --- | --- | --- |
| Resource group | `rg-atomicly-pr-<pr>-<sha7>` | Glob-matchable by the reaper. PR number first so a quick `az group list` is human-scannable. |
| Container Apps Environment | `cae-atomicly-pr-<pr>` | One per PR. SHA is omitted because the environment can be reused across re-deploys of the same PR (cheaper, faster). |
| Container App | `ca-atomicly-pr-<pr>` | Same reason as the environment. Image tag carries the SHA. |
| Postgres Flexible Server | `psql-atomicly-pr-<pr>-<sha7>` | Must be globally DNS-unique; SHA disambiguates re-creates after a force-push. Max 63 chars: `psql-atomicly-pr-9999-abcdefg` = 28 chars. ✅ |
| Key Vault | `kv-atompr-<pr>-<sha7>` | KV is 3–24 chars, alnum + hyphens, globally unique. `kv-atompr-9999-abcdefg` = 22 chars. ✅ |
| ACR repo + tag | `cratomiclypreview<suffix>.azurecr.io/atomicly:pr-<pr>-<sha7>` | Uses a **dedicated shared preview ACR** in `rg-atomicly-preview-shared`, provisioned once by `pr-preview-bootstrap.yml`. NOT the dev ACR — touching the dev ACR (even just a role assignment) would violate the "no dev RG modification" requirement. Tags are cheap; the reaper cleans stale tags as housekeeping (Phase 2). |
| Container App URL | Auto-generated: `https://ca-atomicly-pr-<pr>.<env-hash>.<region>.azurecontainerapps.io` | The `<env-hash>` is generated by Azure when the Container Apps Environment is created and is **not** predictable PR-side. The deploy job MUST output the actual FQDN to `$GITHUB_OUTPUT`; downstream jobs and the PR comment use that value. The teardown workflow finds resources by tag, never by URL. |

**Rules:**
- `<pr>` = `${{ github.event.pull_request.number }}` (always numeric).
- `<sha7>` = first 7 chars of `${{ github.event.pull_request.head.sha }}`.
- The full **resource name** (RG, Postgres, KV, etc.) MUST be derivable from those two inputs alone, so the teardown workflow can compute names without reading state. The **URL** is NOT derivable; it is reported by the deploy job's output.
- Name length: `psql-atomicly-pr-<pr>-<sha7>` = 18 + len(pr) + 1 + 7 chars; safe up to 7-digit PR numbers (max length 63). `kv-atompr-<pr>-<sha7>` = 16 + len(pr); safe up to 8-digit PR numbers (max 24). A workflow preflight rejects PR numbers ≥ 10⁶ with a clear error.

### Decision 2: Workflow topology

Three new workflows, no changes to existing ones.

#### `pr-preview.yml` — deploy + test

- **Trigger:** `pull_request` types `[opened, reopened, synchronize, ready_for_review]`, `branches: [master]`. `concurrency: group: preview-${{ github.event.pull_request.number }}, cancel-in-progress: true` so a force-push cancels the in-flight deploy and starts the new one.
- **Skip rule:** if `github.event.pull_request.draft == true`, exit 0 with a status of "skipped"; previews on drafts cost real money. Re-run on `ready_for_review`.
- **Fork rule:** if `github.event.pull_request.head.repo.full_name != github.repository`, exit with skipped status. Fork PRs cannot obtain OIDC tokens with the preview scope and would silently fail.
- **Pre-deploy stale cleanup:** the deploy job's first step lists `az group list --tag pr=<pr>` and deletes any matching RG whose `commit` tag is not the current SHA. This catches partial RGs left behind by a `cancel-in-progress` cancellation on a previous push.
- **Jobs (sequential happy-path, with `if: always()` cleanup):**
  1. `validate` — same as master CI: `typecheck`, `lint`, `test:run`, `build`. Cheap; fails before we touch Azure.
  2. `build-image` — `docker buildx build --target runner` and `--target migrator`, push to the **shared preview ACR** with tag `pr-<pr>-<sha7>` and a moving `pr-<pr>-latest`.
  3. `deploy` — pre-deploy cleanup (above), then `az group create` (tags applied here, `created-at` computed once in this step), then `az deployment group create --resource-group rg-atomicly-pr-<pr>-<sha7> --template-file infra/preview.bicep`. Outputs the Container App FQDN.
  4. `migrate` — `docker run --rm` the migrator image with `DATABASE_URL` pointing at the freshly-provisioned Postgres. Postgres firewall rule for the runner IP is added before, removed in `trap EXIT`.
  5. `wait-healthy` — poll the Container App FQDN's `/api/healthz` with exponential backoff for up to **5 minutes** (cold start on Container Apps after a fresh environment + image pull can take 2–3 min; 60 s is not enough).
  6. `playwright` — `npm run test:e2e` with `BASE_URL` set to the Container App FQDN. Uploads traces + screenshots as a workflow artifact on failure. **This step is the gate.** Failure here marks the PR check red even if every prior step succeeded.
  7. `report` — `if: always()` — `actions/github-script` posts/updates a single bot comment with: preview URL, Playwright pass/fail count (or "deploy failed" if earlier jobs failed), teardown ETA, link to the run, link to artifacts. Runs whether earlier jobs passed, failed, or were cancelled.
- **Failure paths:**
  - `validate` fails → no Azure resources created. PR check red.
  - `build-image` fails → no Azure resources created. PR check red.
  - `deploy` fails → resource group may exist with partial resources. The `closed` event will still tear it down; reaper covers any case where the PR is force-merged before close.
  - `migrate` fails → the preview is unhealthy; Playwright will fail. PR check red.
  - `playwright` fails → PR check red. **Resources stay up** until PR closes so the developer can debug. Reaper enforces the 24 h ceiling.

#### `pr-preview-teardown.yml` — close-driven teardown

- **Trigger:** `pull_request` types `[closed]`, `branches: [master]`. Runs whether merged or not.
- **Job:**
  1. OIDC login as the `pr-preview` principal.
  2. `RG=$(az group list --tag pr=${{ github.event.pull_request.number }} --query "[?starts_with(name, 'rg-atomicly-pr-')].name" -o tsv)`.
  3. For each `RG`: `az group delete --name "$RG" --yes --no-wait` (Azure CLI does NOT accept wildcards; we always resolve to exact names first).
  4. KV name reuse: because Standard KV keeps a 7-day soft-delete shadow even with purge protection off, run `az keyvault purge --name kv-atompr-<pr>-<sha7> --no-wait` after the RG delete so the next push to the same PR can re-create the KV under the same name.
- **Idempotent.** RG may not exist; KV may not be in soft-delete; both cases are tolerated.

#### `pr-preview-reaper.yml` — hourly safety-net cron

- **Trigger:** `schedule: cron: '0 * * * *'` (every hour on the hour) + `workflow_dispatch` with a `dryRun: boolean` input (default `true` on manual runs, `false` on schedule). Hourly because a daily cron would push the worst-case teardown window to ~48 h, breaking the spec's 24 h requirement.
- **Job:**
  1. OIDC login (federated subject `repo:...:ref:refs/heads/master`, since cron runs from `master`, not `pull_request`).
  2. `az group list --tag lifetime=ephemeral -o json`.
  3. For each RG, classify:
     - **Delete** if: name matches `^rg-atomicly-pr-[0-9]+-[a-f0-9]{7}$` AND `created-by=github-actions` AND (`created-at` > 24 h ago OR the named PR is closed).
     - **Quarantine + alert** if: name matches the prefix but is missing required tags OR has the lifetime tag but does NOT match the prefix. Emit a workflow-error annotation and (Phase 2) page the owner. Do NOT delete.
     - **Skip** if: tag is present but the RG is < 24 h old and the PR is still open.
  4. KV purge: for each deleted RG, also purge any soft-deleted KV matching `kv-atompr-*` whose deletion date is older than 6 days (last-chance purge before the 7-day soft-delete window expires).
  5. Summary step: print counts of inspected / deleted / skipped / quarantined. Fail the workflow run if any quarantined items were found.
- **Safety:** double guard. Both the name-prefix regex AND the `lifetime=ephemeral` tag must match before any delete. The Azure Policy deny rule (Decision 8) is a third layer that prevents creation of any RG outside the prefix in the first place.

### Decision 3: Bicep parameter strategy — one parameter set, suffix is the discriminator

`infra/preview.bicep` is a **resource-group-scoped** template (vs `main.bicep` which is sub-scoped) so deploy + teardown can use `az group create` / `az group delete` rather than subscription-level deployments.

```bicep
@description('Pull request number. Used in every resource name and tag.')
param prNumber string

@description('Short commit SHA (7 chars).')
param commitSha string

@description('Location.')
param location string = resourceGroup().location

@description('Postgres admin password (generated fresh per run by the workflow, passed via @secure parameter, never echoed or stored).')
@secure()
param postgresAdminPassword string

@description('Image tag to deploy (e.g., pr-1234-abcdef0).')
param imageTag string

@description('Shared preview ACR login server (cratomiclypreview<suffix>.azurecr.io).')
param acrLoginServer string

@description('ISO-8601 UTC timestamp of preview creation. Set ONCE by the workflow on the first deploy of this PR + SHA; passed in unchanged on re-deploys so the reaper clock does not reset.')
param createdAt string

var commonTags = {
  pr: prNumber
  commit: commitSha
  lifetime: 'ephemeral'
  'created-by': 'github-actions'
  'created-at': createdAt
}
```

All names are derived from `prNumber` and `commitSha`. No environment toggle — preview is a single shape. Modules consumed: `containerApp`, `postgres` (B1ms, 20 GB, autoGrow off, no HA, 7-day backup), `keyvault` (Standard, soft-delete on — required by Azure, can't be disabled — purge protection **off**), `monitoring` (0.1 GB/day cap, 7-day retention), `acrPull`.

**Key Vault purge protection is explicitly OFF on previews.** Soft-delete cannot be disabled (Azure now mandates it), but purge protection adds an additional 7–90-day undeletable shadow. With purge protection off, the teardown workflow runs `az keyvault purge` after RG deletion so the next push to the same PR can re-create the KV under the same name. Without that purge, a force-push (which changes `<sha7>`) would force a new KV name; not a correctness problem, but it creates orphan soft-deleted KVs that count against the per-region KV name quota.

**`postgresAdminPassword` is generated fresh per workflow run** via `openssl rand -base64 32`, passed to Bicep as a `@secure()` parameter, and stored only in the preview KV. It is never written to a GitHub secret. The "no stored Azure credentials" requirement refers to long-lived auth credentials (client secrets, API keys); per-run-generated database passwords passed through `@secure` parameters are explicitly permitted.

### Decision 4: Database migration on a fresh server per PR

Each preview gets its own Postgres Flexible Server. Migrations run as a one-shot Docker container in the workflow:

1. Bicep finishes provisioning Postgres.
2. Workflow adds the runner IP to the Postgres firewall (`az postgres flexible-server firewall-rule create`). `trap EXIT` removes it.
3. `docker run --rm -e DATABASE_URL=... atomicly-migrator:pr-<pr>-<sha7>` runs `npm run prisma:migrate:deploy`.
4. No seed data. The Playwright suite is responsible for creating any state it needs through the public app surface (sign-up flow, etc.).

**Rationale for fresh-server-per-PR (not shared preview Postgres):**
- Schema-divergent PRs (a column rename + a column add on parallel branches) would corrupt a shared DB.
- B1ms is cheap enough that isolation wins over reuse.
- "Worked on my preview" is only meaningful if the preview is fully isolated.

### Decision 5: Teardown + 24 h reaper

Two-layer defence:
- **Layer 1 (close-driven):** `pr-preview-teardown.yml` runs immediately on PR close. ~95 % of cases.
- **Layer 2 (cron):** `pr-preview-reaper.yml` runs daily, deletes anything older than 24 h regardless of PR state.

The reaper exists because:
- A workflow can fail to fire (GitHub Actions outage, repo archived mid-PR, force-merge bypassing close webhook).
- A `closed` event on a PR opened from a **fork** runs with `GITHUB_TOKEN` permissions that may not have OIDC access. The reaper runs from the base repo on a schedule, so it always has full credentials.
- Operator confidence: "if I forget about a preview, it dies in ≤24 h" is a much stronger guarantee than "if I forget, it dies eventually."

### Decision 6: Cost guardrails

**SKU table for one preview (Australia East, list price retrieved 2026-06-18 from Azure Retail Prices API, ~24 h window):**

| Resource | SKU / config | Idle $/day | Worst-case $/day |
| --- | --- | --- | --- |
| Postgres Flexible | B1ms, 20 GB, autoGrow off, no HA, 7-day backup | $0.62 compute + $0.09 storage = **$0.71** | $0.71 |
| Container App | Consumption, minReplicas 0, maxReplicas 2, 0.5 vCPU / 1 GiB | $0.00 | $0.05 (1 h Playwright run) |
| Container Apps Environment | Consumption-only, no dedicated workload profile | $0.00 | $0.00 |
| Log Analytics | 0.1 GB/day cap, 7-day retention, AUE @ $2.99/GB | $0.00 (no traffic = no ingestion) | $0.30 (cap × ingest price) |
| Key Vault | Standard, ≤ 100 ops/day | $0.00 | $0.00 |
| Shared preview ACR | Basic, amortised across all open previews | ~$0.00 marginal | ~$0.00 |
| **Total** | | **~$0.72/day** | **~$1.06/day** |

The Log Analytics daily cap was lowered from the originally-drafted 0.2 GB/day to 0.1 GB/day because 0.2 GB × $2.99 = $0.60/day, which on its own pushed the worst-case above the $1/day target. 0.1 GB is still ~50× the typical idle ingestion measured on the dev workspace.

**"If a preview RG is forgotten for 30 days, what is the worst-case bill?"**
- With the **hourly** reaper working: the reaper deletes anything > 24 h, so worst case is 25 h × $1.06 = **~$1.10**.
- With the reaper broken AND the close webhook missed for 30 days: 30 × $1.06 = **~$32**. The Phase-2 subscription budget alert at $50/mo with email at 50 %/80 %/100 % catches this within ~2 weeks even if the cron is silently failing.
- With the Azure Policy deny rule also somehow bypassed (extremely unlikely; would require disabling the policy assignment): bounded only by subscription quota — Postgres server quota of 20 caps blast radius at ~$640/mo absolute worst case.

**Budget alert idea (out of scope to implement here, in scope to design):**
Configure an Azure subscription budget at $50/mo with an action group that emails the owner at 50 %, 80 %, 100 %.

**Tag-based query to list every active preview and its hourly burn:**
```bash
# List active previews, their PR, age, and Postgres SKU.
az group list \
  --tag lifetime=ephemeral \
  --query "[?starts_with(name, 'rg-atomicly-pr-')].{name:name, pr:tags.pr, created:tags.\"created-at\"}" \

```

### Decision 7: Security — locking down a preview without Front Door

Previews are private testbeds. We don't need WAF / DDoS / public CDN. We DO need to ensure they aren't accidentally indexable.

**Origin lockdown for previews uses Container Apps IP restrictions:**

- Default action: `Deny`.
- Allowed IPs:
  - **The current GitHub Actions runner's public egress IP**, resolved at job start with `curl -s https://api.ipify.org`. One IP, one rule. Removed in a `trap EXIT` cleanup at the end of the job. The GitHub-published `actions[]` ranges are NOT used: there are ~3000 entries (changes weekly), they collide with the Container Apps 50-IP-rule limit, and they change mid-month frequently enough to break long-running deploys.
  - **The Playwright job's runner public IP**, added at the start of the Playwright job and removed on exit. (Each job in a GitHub workflow runs on a different runner with a different IP, so each job that talks to the preview must add/remove its own IP.)
  - **Optional reviewer IP**, added by a *separate* `pr-preview-open.yml` `workflow_dispatch` workflow that accepts `pr_number` and `reviewer_ip` as inputs. Auto-removes after 4 h via a scheduled follow-up job, or sooner when the reviewer re-runs with `--revoke`. `pull_request` events cannot accept arbitrary `inputs`, so this MUST be a separate workflow.
- App-layer controls inherit from the existing codebase: CSP nonce, security headers, in-memory rate limit, CSRF guard, timing-safe auth — all already in `proxy.ts` / `lib/security/*`.
- **No Front Door, no Cloudflare, no WAF.** The threat model on a private testbed is "don't accidentally end up on shodan", not "survive a determined attack". IP-allowlist + auth is sufficient.

### Decision 8: OIDC scoping and blast-radius isolation

A **separate** Azure AD app + service principal — `pr-preview` — is provisioned for preview workflows. It is NOT the existing dev/master principal. Two federated credential subjects on the new app:

- `repo:vngo4590/atomic-habit-tracker:pull_request` — used by `pr-preview.yml` and `pr-preview-teardown.yml`.
- `repo:vngo4590/atomic-habit-tracker:ref:refs/heads/master` — used by `pr-preview-reaper.yml`, because scheduled and `workflow_dispatch` runs do NOT have the `pull_request` subject claim; they have `ref:` claims tied to whatever branch the workflow file lives on. Without this credential, the reaper cannot log in.

**Authorisation strategy: subscription-scoped Contributor + Azure Policy deny.**

ABAC tag conditions on RBAC role assignments were considered and rejected: Azure ABAC for `Microsoft.Resources/subscriptions/resourceGroups/write` is not GA / not reliable for *gating creation by requested-tag values* (it works for storage/KV data plane and a small set of other resource providers, not for ARM resource creation). The rubber-duck reviewer correctly flagged this.

Instead:

1. The `pr-preview` principal is assigned `Contributor` at the subscription scope.
2. An **Azure Policy assignment** at subscription scope blocks any resource group create whose name does NOT match `^rg-atomicly-pr-[0-9]+-[a-f0-9]{7}$` OR which lacks the tags `lifetime=ephemeral` and `created-by=github-actions`. Effect: `Deny`. The policy applies to ALL principals, but only this one creates RGs in normal operation.
3. A second policy assignment forbids any role assignment changes outside `rg-atomicly-pr-*` and `rg-atomicly-preview-shared` by the `pr-preview` principal.

Net effect: even with subscription-scoped Contributor, the `pr-preview` principal cannot create or modify a non-preview resource group. The dev RG is unreachable.

The existing dev/master principal is **unchanged**. Master CI/CD continues to authenticate as before.

### Decision 9: Quota preflight + active-preview ceiling

Azure subscription defaults that bite at low double-digit preview counts:

- **Postgres Flexible Server count: 20 per subscription per region** (default; raisable on request).
- **Container Apps Environment count: 15 per subscription per region** (default).
- **Key Vault names per region: large but soft-deleted vaults count for 7+ days** after deletion.

To avoid an unrelated PR failing because someone else's preview consumed quota, the `pr-preview.yml` `deploy` job runs a preflight:

```bash
ACTIVE=$(az group list --tag lifetime=ephemeral --query "length([?starts_with(name, 'rg-atomicly-pr-')])" -o tsv)
HARD_CAP=10
if [ "$ACTIVE" -ge "$HARD_CAP" ]; then
  echo "::error::Active preview cap ($HARD_CAP) reached. $ACTIVE previews are live. Close or merge older PRs, or wait for the reaper."
  exit 1
fi
```

The cap (10) is half of the Postgres quota so concurrent re-deploys under `cancel-in-progress` still have headroom. Raising the cap requires raising the Postgres quota first.

## Risks / Trade-offs

- **[Postgres provisioning is slow — typically 3–5 min, sometimes 10]** → Mitigation: run Bicep deploy + image build in parallel where possible; accept that PR previews take ~6–8 min to come up the first time. Re-deploys are faster because the Container App + KV + LA are reused.
- **[Per-PR Postgres = lots of small servers]** → Subscription quota for Postgres servers is 20 by default. Decision 9 caps active previews at 10. File a quota-bump ticket if usage approaches the cap.
- **[GitHub-fork PRs can't authenticate to Azure]** → Mitigation: `pr-preview.yml` checks `github.event.pull_request.head.repo.full_name == github.repository` and skips on forks. Document this clearly in `infra/README.md`. Forks fall back to standard CI only.
- **[Tag-based reaping is the only thing protecting cost]** → Mitigation: belt-and-braces. The reaper requires BOTH the name prefix AND the lifetime tag. The Azure Policy deny rule (Decision 8) prevents creation of any non-conforming RG in the first place. The budget alert (Phase 2) is a fourth layer.
- **[A flaky Playwright test marks every PR red]** → Tradeoff: this is a feature. Flaky preview tests means flaky production tests; we'd rather know on the PR than after merge.
- **[Naming collisions across PRs that re-use a number]** → Not possible; GitHub PR numbers are monotonic and never reused.
- **[Cold-start on first Playwright request after deploy]** → Mitigation: dedicated `wait-healthy` job polls `/api/healthz` with exponential backoff for up to **5 minutes** before Playwright runs. Container Apps cold start after a fresh environment provision can take 2–3 min; 60 s is not enough.
- **[`utcNow()` in Bicep would re-evaluate on re-deploy and reset the reaper clock]** → Mitigation: Bicep declares `param createdAt string` and never calls `utcNow()`. The workflow computes the value once when the RG is created (`az group create --tags ... created-at=$(date -u +...)`) and passes the same value to every subsequent `az deployment group create` for that RG.
- **[Force-push leaves partial RG from the cancelled run]** → Mitigation: Decision 2's pre-deploy stale-cleanup step deletes any RG tagged with the same `pr` but a different `commit` before provisioning the new one. The reaper is a final safety net.
- **[Container Apps IP-restriction rules have a hard limit]** → Mitigation: Decision 7 uses a single per-job runner IP, not the full GitHub Actions range. We're well under any rule cap.
- **[KV soft-delete shadow blocks name reuse on the next push]** → Mitigation: `pr-preview-teardown.yml` runs `az keyvault purge` immediately after `az group delete`. The reaper also purges any orphan soft-deleted KV older than 6 days.
- **[Azure Policy assignment is a manual one-time setup outside this repo's IaC]** → Tradeoff: the policy definition + assignment SHOULD be checked in under `infra/policies/` and applied by the bootstrap workflow, but the **assignment** is sub-scoped and may require an existing Owner to bless. Documented in `infra/README.md` as a one-time human step.

## Migration / Rollout Plan

This is a phased rollout. Phase 1 lands an MVP that covers every spec requirement; Phase 2 hardens.

**Phase 0 (one-time bootstrap, human-driven, documented):**
1. Provision a new Azure AD app + service principal named `pr-preview-atomicly`.
2. Add federated credential subjects: `repo:vngo4590/atomic-habit-tracker:pull_request` and `repo:vngo4590/atomic-habit-tracker:ref:refs/heads/master`.
3. Assign `Contributor` at subscription scope.
4. Assign the two Azure Policy definitions (RG-name + tag, role-assignment-scope) at subscription scope.
5. Run `.github/workflows/pr-preview-bootstrap.yml` (workflow_dispatch) to create `rg-atomicly-preview-shared` with the shared ACR.
6. Store `AZURE_PREVIEW_CLIENT_ID` as a GitHub Actions secret.

**Phase 1 (lands together, this change):**
1. Branch: `feat/ephemeral-pr-preview-envs-phase-1`.
2. Author `infra/preview.bicep` and `az bicep build` it locally.
3. Author the four workflows (`pr-preview.yml`, `pr-preview-teardown.yml`, `pr-preview-reaper.yml`, `pr-preview-bootstrap.yml`).
4. Author the optional reviewer-IP workflow `pr-preview-open.yml`.
5. Author the two Azure Policy JSONs under `infra/policies/`.
6. Open a self-test PR; observe the full deploy + Playwright + teardown loop.
7. Merge.

**Phase 2 (follow-up):**
- Add an Azure Monitor budget alert at $50/mo with email notification.
- Add a `workflow_dispatch` input to `pr-preview-reaper.yml` for `dryRun` and a `force` option.
- Add a Playwright trace+screenshot upload to a public artifact on failure so reviewers don't need to re-run.
- Add ACR-tag housekeeping to the reaper.

**Rollback:**
- Phase 1 rollback: delete the four new workflow files, delete `infra/preview.bicep`, manually run the reaper once with a low age threshold (or delete `rg-atomicly-pr-*` manually), then delete `rg-atomicly-preview-shared` and the `pr-preview` principal. ~15 min. Master CI/CD is untouched throughout.

## Open Questions

- Should preview workflows run on draft PRs? Default: no (cost). Override: a `preview` label triggers it manually. Decision deferred to Phase 1 implementation.
- Should we reuse the dev Key Vault for preview secrets (e.g., AUTH_SECRET) instead of provisioning a per-PR KV? Reuses save ~30 s of provisioning but cross-couples preview lifecycle to dev. Default: per-PR KV.
- Should the reaper also delete dangling ACR tags older than 24 h? Tags are cheap but accumulate. Defer to a separate housekeeping change.
